VLANs (Virtual Local Area Networks) are the primary mechanism CHIP uses to enforce network segmentation. By separating traffic into isolated Layer 2 domains, you ensure that a device in one zone — an IoT sensor, a staging VM, or a publicly-reachable web server — cannot directly communicate with devices in another zone unless an explicit firewall rule permits it. Without VLANs, a single flat network means that any device with a network connection can attempt to reach any other device, which dramatically expands the attack surface and makes security policy nearly impossible to audit. This page provides the authoritative VLAN reference for the CHIP platform, explains how VLANs map to network zones, and documents how ports should be configured on managed switches.

## VLAN Reference Table

The table below is the definitive record of all VLANs in the CHIP network. Every managed switch must be configured with this schema. Any VLAN ID not listed here is not authorized for use and should be removed from switch configurations during audits.

| VLAN ID | Name | Subnet | Gateway | Purpose | Notes |
|---|---|---|---|---|---|
| **10** | Management | 10.0.10.0/24 | 10.0.10.1 | Infrastructure control plane — hypervisors, switches, IPMI, router UI | Strictly restricted; no user devices |
| **20** | Trusted LAN | 10.0.20.0/24 | 10.0.20.1 | User workstations, laptops, personal trusted devices | Default zone for managed user endpoints |
| **30** | IoT | 10.0.30.0/24 | 10.0.30.1 | Smart home, sensors, cameras, embedded devices | Outbound-only to approved endpoints |
| **40** | DMZ | 10.0.40.0/24 | 10.0.40.1 | Internet-facing services behind reverse proxy | No direct inbound from internet; NAT via firewall |
| **50** | Storage | 10.0.50.0/24 | 10.0.50.1 | NFS/iSCSI data plane between hypervisors and storage nodes | Hypervisor and storage nodes only |
| **99** | Native | — | — | Default/untagged VLAN; no routing | Quarantine zone for unconfigured ports |

<Info>
  Gateway addresses follow a consistent `.1` convention within each subnet (e.g., 10.0.10.**1**, 10.0.20.**1**). The firewall holds the gateway interface for every VLAN and is the sole routing point between segments.
</Info>

## VLAN Details

<Tabs>
  <Tab title="VLAN 10 — Management">
    The Management VLAN is the most privileged zone in the platform. It carries all infrastructure control-plane traffic: Proxmox web UI and API, switch management interfaces, IPMI/iDRAC out-of-band management, firewall administration, and DNS/DHCP server access.

    **Access policy:** Only explicitly authorized administrative hosts may communicate on this VLAN. No user devices, IoT devices, or general-purpose VMs should have interfaces in this zone.

    **DHCP:** Static reservations only. No dynamic leases are issued on VLAN 10. Every device in this zone has a fixed IP defined in the DHCP reservation table.

    **Services homed here:**
    - `router-01` — 10.0.10.1
    - `sw-core-01` — 10.0.10.2
    - `sw-access-01` — 10.0.10.3
    - `pve-node-01` — 10.0.10.11
    - `pve-node-02` — 10.0.10.12
    - `pve-node-03` — 10.0.10.13
    - `dns-01` — 10.0.10.20
    - `vault-01` — 10.0.10.21
    - `monitoring-01` — 10.0.10.22
  </Tab>

  <Tab title="VLAN 20 — Trusted LAN">
    The Trusted LAN is the primary zone for user-operated devices: workstations, laptops, tablets, and other devices owned and controlled by trusted users. Devices here can browse the internet, reach DMZ-hosted services, and consume internal platform services like DNS and NTP.

    **Access policy:** Devices in this zone cannot initiate connections to the Management VLAN. Inbound connections from IoT or DMZ to this zone are blocked by default.

    **DHCP:** Dynamic leases issued from a pool of 10.0.20.100–10.0.20.254. Known devices should have static reservations in the 10.0.20.10–10.0.20.99 range.

    **Typical residents:** Workstations, personal laptops, development machines, printers
  </Tab>

  <Tab title="VLAN 30 — IoT">
    The IoT VLAN isolates smart home devices, IP cameras, sensors, smart plugs, media players, and any other embedded device you do not fully trust at the software level. These devices often run outdated firmware, phone home to vendor cloud services, and occasionally have known vulnerabilities — keeping them segregated limits what they can do if compromised.

    **Access policy:** IoT devices can reach the internet (restricted to approved outbound endpoints via firewall policy), but cannot reach the Trusted LAN, Management, or Storage VLANs. IoT devices cannot reach each other by default — consider enabling intra-VLAN isolation on the access switch if supported.

    **DHCP:** Dynamic leases from 10.0.30.100–10.0.30.254. Smart home hubs that act as gateways for sub-devices should have static reservations.

    **Typical residents:** Smart speakers, IP cameras, smart switches/plugs, thermostats, media streamers
  </Tab>

  <Tab title="VLAN 40 — DMZ">
    The DMZ (Demilitarized Zone) hosts services that need to be reachable from the internet or that serve as the public face of internally-hosted applications. The reverse proxy (`proxy-01`) sits here and terminates incoming HTTPS connections before forwarding traffic to upstream services on other VLANs.

    **Access policy:** Traffic from the internet reaches the DMZ only through NAT rules on the firewall — no direct inbound connections to any other zone. DMZ hosts can reach specific upstream services on the Management or Storage VLANs only via explicit firewall rules. DMZ hosts cannot initiate connections to the Trusted LAN.

    **DHCP:** Static reservations only. All DMZ hosts have fixed IPs.

    **Typical residents:** `proxy-01` (reverse proxy), public-facing APIs, self-hosted web services, SMTP relays
  </Tab>

  <Tab title="VLAN 50 — Storage">
    The Storage VLAN is dedicated to high-throughput, low-latency block and file storage traffic. By isolating storage traffic onto its own VLAN and physical/logical network path, you prevent NFS or iSCSI transfers from saturating the same network segments that carry user traffic or VM management communications.

    **Access policy:** Only hypervisor nodes and designated storage hosts have interfaces on this VLAN. No user devices, containers, or internet-facing hosts belong here.

    **DHCP:** Static reservations only. Storage paths must be predictable.

    **Jumbo frames:** MTU 9000 (jumbo frames) is recommended for all devices on VLAN 50 to maximize NFS/iSCSI throughput. Ensure the switch port MTU is also configured to support this.

    **Typical residents:** `storage-01` (NAS/NFS), `pve-node-*` (storage interface)
  </Tab>

  <Tab title="VLAN 99 — Native">
    VLAN 99 is the designated Native VLAN for all 802.1Q trunk links. It carries no routed traffic and has no associated subnet or gateway. Its purpose is to ensure that any untagged frames that land on a trunk port (from a misconfigured or non-VLAN-aware device) are captured in a dead-end segment rather than silently placed into a production VLAN.

    **Access policy:** No routing. No DHCP. Complete isolation.

    **Port usage:** VLAN 99 is set as the native (untagged) VLAN on all trunk ports. Access ports that are not yet assigned to a zone should be placed in VLAN 99 rather than left in a default VLAN that may carry production traffic.

    <Warning>
      Never set a production VLAN as the native VLAN on a trunk port. Doing so would allow untagged traffic — including traffic from non-VLAN-aware devices — to silently enter that production segment.
    </Warning>
  </Tab>
</Tabs>

## Trunk vs. Access Ports

Every switch port in CHIP is configured in one of two modes. Understanding the difference is essential for correct switch configuration.

### Access Ports

An access port belongs to exactly one VLAN. Frames entering the port are untagged, and the switch assigns them to the configured VLAN. Frames leaving the port have their VLAN tag stripped — the connected device never sees the tag.

Use access ports for:
- End-user workstations and laptops (VLAN 20)
- IoT devices and smart home hardware (VLAN 30)
- Printers and single-purpose appliances
- Any device that is not VLAN-aware

```bash
# Example: Configuring an access port on a Linux bridge (for testing)
ip link add link eth0 name eth0.20 type vlan id 20
ip link set eth0.20 up
ip addr add 10.0.20.5/24 dev eth0.20
```

### Trunk Ports

A trunk port carries multiple VLANs simultaneously, with each frame tagged with its VLAN ID using the 802.1Q standard. The connected device must be VLAN-aware and capable of creating sub-interfaces or virtual bridges per VLAN.

Use trunk ports for:
- Uplinks between switches
- Hypervisor node uplinks (must carry all VLANs the node's VMs use)
- Firewall LAN interface (must carry all defined VLANs)

<Note>
  All Proxmox hypervisor nodes must be connected to trunk ports carrying **at a minimum** VLANs 10, 20, 40, and 50. VLAN 30 should be included if IoT VMs or bridge containers are hosted on that node. VLAN 99 must be set as the native VLAN on these ports. Configure the Linux bridge on each Proxmox node with VLAN-aware mode enabled so that individual VMs and containers can be assigned to specific VLANs via their network interface configuration in the Proxmox UI.
</Note>

## VLAN Tagging in Proxmox

When you create a virtual machine or LXC container in Proxmox and want to place it on a specific VLAN, you assign the VLAN tag directly to the VM's network interface. Proxmox handles the 802.1Q tagging at the hypervisor bridge level — the VM itself does not need to be VLAN-aware.

```bash
# Proxmox VM network configuration (in /etc/pve/qemu-server/<vmid>.conf)
# Assign VM to VLAN 20 (Trusted LAN) via the vmbr0 bridge
net0: virtio=AA:BB:CC:DD:EE:FF,bridge=vmbr0,tag=20

# Assign LXC container to VLAN 40 (DMZ)
# In /etc/pve/lxc/<ctid>.conf
lxc.net.0.vlan.id = 40
```

Alternatively, configure this through the Proxmox web UI by setting the **VLAN Tag** field on the network device tab of any VM or container. The bridge (`vmbr0`) should be configured in VLAN-aware mode on the Proxmox host:

```bash
# /etc/network/interfaces on Proxmox node
auto vmbr0
iface vmbr0 inet manual
    bridge-ports eno2
    bridge-stp off
    bridge-fd 0
    bridge-vlan-aware yes
    bridge-vids 2-4094
```

<Tip>
  Use a dedicated bridge interface (`vmbr0`) for your VLAN-tagged VM traffic and a separate management bridge (`vmbr1`) or a direct interface for the Proxmox node's own management IP on VLAN 10. This prevents accidental misconfiguration of the management interface when modifying VM networking.
</Tip>

## Inter-VLAN Routing Policy

As described in the [Network Overview](/network/overview), inter-VLAN routing occurs exclusively at the firewall. The following routing rules summarize the permitted traffic flows between VLANs:

| Source VLAN | Destination VLAN | Permitted | Protocol/Port | Rule Rationale |
|---|---|---|---|---|
| VLAN 10 (Mgmt) | All VLANs | ✅ Yes | Any | Admins need unrestricted operational access |
| VLAN 20 (Trusted) | Internet | ✅ Yes | TCP 80, 443 | Standard user web traffic |
| VLAN 20 (Trusted) | VLAN 40 (DMZ) | ✅ Yes | TCP 443 | Access to internal services via HTTPS |
| VLAN 20 (Trusted) | VLAN 10 (Mgmt) | ❌ No | — | Prevents workstations from reaching infra UIs |
| VLAN 30 (IoT) | Internet | ✅ Restricted | Vendor endpoints | Allow firmware updates and telemetry |
| VLAN 30 (IoT) | All internal VLANs | ❌ No | — | Full isolation of IoT devices |
| VLAN 40 (DMZ) | Internet | ✅ Yes | TCP 80, 443 | Services call external APIs |
| VLAN 40 (DMZ) | VLAN 20 (Trusted) | ❌ No | — | Published services must not reach user LAN |
| VLAN 40 (DMZ) | VLAN 10 (Mgmt) | ✅ Limited | TCP 443 | Reverse proxy may reach specific backend services |
| VLAN 50 (Storage) | VLAN 10 (Mgmt) | ✅ Yes | TCP 2049, 3260 | NFS/iSCSI from hypervisors |
| VLAN 50 (Storage) | All others | ❌ No | — | Storage traffic stays on storage path |

<Warning>
  Adding firewall rules that permit traffic from VLAN 30 (IoT) or VLAN 40 (DMZ) into VLAN 20 (Trusted LAN) bypasses the core security model of the platform. If you believe you need such a rule, pause and reconsider whether the workload is placed in the correct zone rather than punching a hole in the segmentation.
</Warning>

## Adding a New VLAN

If you need to introduce a new network segment, follow these steps to maintain consistency across the platform:

<Steps>
  <Step title="Define the VLAN in the repository">
    Add the new VLAN to the network configuration file in the repository. Assign an unused VLAN ID, choose a subnet in a non-overlapping range, and document the zone purpose and access policy before committing.
  </Step>
  <Step title="Configure the VLAN on managed switches">
    Add the new VLAN ID to the core switch and any access switches that need it. Add the VLAN to trunk ports that should carry it (typically all uplinks and hypervisor ports). Open a change record for this work.
  </Step>
  <Step title="Create firewall interfaces and rules">
    Add a new interface on the firewall for the VLAN, assign the gateway IP, and configure the DHCP scope if dynamic addressing is needed. Define the inter-zone firewall rules — starting from a default-deny baseline and adding only what is required.
  </Step>
  <Step title="Update the CHIP inventory">
    Add the new VLAN to the network zone table in the inventory. Update any hosts that will have interfaces in the new VLAN.
  </Step>
  <Step title="Validate end-to-end">
    Place a test device in the new VLAN, confirm it receives an address (or accepts a static assignment), verify DNS resolution works, and confirm that traffic to other zones obeys the firewall policy you defined. Run a quick nmap scan from the new VLAN toward protected zones to validate deny rules are in effect.
  </Step>
</Steps>
