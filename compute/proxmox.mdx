CHIP runs its compute layer on **Proxmox VE 8.x**, an open-source hypervisor platform that combines KVM-based virtual machines and LXC containers under a single management plane. The cluster spans multiple physical nodes distributed across availability zones, providing redundancy for both workloads and the management control plane itself. Understanding the cluster topology, quorum rules, and access model is essential before you provision or modify any compute resources.

## Cluster Topology

The CHIP Proxmox cluster is named `chip-pve` and consists of five physical nodes. Three of those nodes carry the **voter** role that participates in quorum decisions; the remaining two are **regular** members that run workloads but do not contribute quorum votes. This arrangement ensures the cluster can tolerate the loss of one quorum voter without losing consensus.

<Note>
  Proxmox clusters require a strict quorum majority to remain operational. If fewer than two of the three quorum-voting nodes are reachable, the cluster enters a **no-quorum** state and all running VMs are suspended to prevent split-brain data corruption. Never power down more than one quorum node simultaneously.
</Note>

## Cluster Configuration

### Corosync & Quorum

Corosync manages all inter-node communication over a dedicated cluster network (`10.10.0.0/24`) on interface `ens3` of each node. This network is isolated from VM traffic and management traffic to prevent interference during high-load events.

```bash
# View current cluster quorum status
pvecm status

# List all cluster nodes and their online/offline state
pvecm nodes
```

The corosync configuration lives at `/etc/pve/corosync.conf` and is replicated automatically by the Proxmox cluster file system (pmxcfs). **Do not edit this file manually.** Use `pvecm` commands or the web UI instead.

### Cluster File System (pmxcfs)

All cluster-wide configuration — VM and container configs, storage definitions, user databases — is stored in pmxcfs, a distributed, database-backed filesystem mounted at `/etc/pve`. Changes written on any node are immediately visible on all other nodes. The filesystem is read-write only when quorum is met; it becomes read-only during a no-quorum event.

## High Availability

CHIP uses Proxmox HA Manager to automatically restart VMs and containers on surviving nodes if a host fails. HA is configured at two levels: **HA Groups** define which nodes a resource is allowed to run on, and **HA Resources** enroll individual VMs or containers for automatic failover.

### HA Groups

| Group Name        | Allowed Nodes                          | No-Failback | Priority |
|-------------------|----------------------------------------|-------------|----------|
| `ha-tier1`        | pve-node-01, pve-node-02, pve-node-03  | false       | 100      |
| `ha-tier2`        | pve-node-04, pve-node-05               | false       | 50       |
| `ha-infra`        | pve-node-01, pve-node-02               | true        | 200      |

To enroll a VM in HA, navigate to **Datacenter → HA → Resources → Add**, choose the VMID, assign an HA group, and set the desired state to `started`.

```bash
# Enroll VM 200 into the ha-tier1 group via CLI
ha-manager add vm:200 --group ha-tier1 --state started

# Check current HA resource states
ha-manager status
```

<Warning>
  Setting a VM's HA desired state to `started` means the cluster will attempt to keep it running at all times, including automatic restarts after node failures. Only enroll VMs that have been validated for stateless or replication-safe operation. Databases and write-heavy workloads must use shared storage (Ceph or NFS) before being enrolled.
</Warning>

## Storage Backends

Proxmox storage is configured per-node and cluster-wide depending on the backend type. See the [Storage](/compute/storage) page for full pool details. The following backends are active on the CHIP cluster:

<CardGroup cols={2}>
  <Card title="local-lvm" icon="database">
    LVM-thin pool on each node's local NVMe. Used for VM disks where live migration is **not** required. High IOPS, node-local.
  </Card>
  <Card title="ceph-rbd" icon="circle-nodes">
    Ceph RBD block storage distributed across all five nodes. Required for HA VMs. Supports live migration and snapshots.
  </Card>
  <Card title="nfs-backup" icon="hard-drive">
    NFS share on `storage-01.chip.internal` used exclusively for Proxmox Backup Server (PBS) targets and ISO images.
  </Card>
  <Card title="zfs-tank" icon="layer-group">
    ZFS pool on pve-node-01 through pve-node-03 used for container root filesystems and secondary VM disks.
  </Card>
</CardGroup>

## Network Bridges

Each Proxmox node exposes a set of Linux bridges that connect VM and container network interfaces to the appropriate physical or VLAN-tagged network.

| Bridge   | VLAN Tag | Subnet              | Purpose                              |
|----------|----------|---------------------|--------------------------------------|
| `vmbr0`  | untagged | 10.0.0.0/24         | Management & Proxmox web UI          |
| `vmbr1`  | 10       | 10.10.0.0/24        | Cluster / Corosync heartbeat         |
| `vmbr2`  | 20       | 10.20.0.0/22        | Production workload network          |
| `vmbr3`  | 30       | 10.30.0.0/22        | Development & staging workload network|
| `vmbr4`  | 40       | 10.40.0.0/24        | Storage / Ceph replication network   |

<Tip>
  Always attach production VMs to `vmbr2` and development workloads to `vmbr3`. Mixing environments on the same bridge undermines network policy enforcement and complicates firewall audits.
</Tip>

## Accessing Proxmox

### Web Interface

The Proxmox management UI is available at `https://proxmox.chip.internal:8006`. You can reach any individual node directly, but the cluster view is identical from all nodes.

<Steps>
  <Step title="Open the web UI">
    Navigate to `https://proxmox.chip.internal:8006` in your browser. Accept the self-signed certificate warning — the certificate is issued by the internal CHIP CA and is safe to trust on the corporate network.
  </Step>
  <Step title="Select the authentication realm">
    On the login screen, set the **Realm** dropdown to `chip-ldap` to authenticate with your corporate LDAP/Active Directory credentials. Use the `pam` realm only for break-glass access with a local node account.
  </Step>
  <Step title="Enter your credentials">
    Type your **username** (without the domain suffix) and **password**, then click **Login**. If your account is new, contact the infrastructure team to have it added to the `PVE-Users` AD group.
  </Step>
  <Step title="Verify your resource pool">
    After login, confirm that the left-hand tree shows your assigned **Resource Pool** (e.g., `pool-platform` or `pool-data`). If the pool is missing, you have landed in a scope that may not show your VMs — notify the infra team.
  </Step>
</Steps>

### API Access

The Proxmox REST API is available at the same base URL. CHIP issues dedicated API tokens for automation tooling so that human credentials are never embedded in scripts.

```bash
# Obtain an API token (done once by an admin)
# Navigate to Datacenter → Permissions → API Tokens → Add

# Example: list all VMs using an API token
curl -s \
  -H "Authorization: PVEAPIToken=automation@pve!chip-ci=<token-secret>" \
  https://proxmox.chip.internal:8006/api2/json/cluster/resources?type=vm \
  | jq '.data[] | {vmid, name, status, node}'
```

<Info>
  API tokens carry the same permission scope as the user they belong to. Always create tokens under a dedicated service account (e.g., `automation@chip-ldap`) rather than a personal account so that token revocation does not break pipelines when personnel change.
</Info>

## Resource Pools & RBAC

Proxmox uses a **path-based permission model**. Permissions are granted as `(user/group, role, path)` triples. Paths correspond to objects in the cluster tree: `/vms/<vmid>`, `/storage/<storeid>`, `/nodes/<nodeid>`, or the root `/`.

### Standard Roles in Use

| Role Name          | Key Privileges                                              | Assigned To                      |
|--------------------|-------------------------------------------------------------|----------------------------------|
| `PVEVMUser`        | Start, stop, console access to VMs                          | Application team members         |
| `PVEVMAdmin`       | Full VM lifecycle: create, delete, snapshot, migrate        | Platform engineers               |
| `PVEDatastoreUser` | Read storage, create VM disks                               | CI/CD service accounts           |
| `PVEAdmin`         | All privileges except modifying cluster-wide config         | Senior infrastructure engineers  |
| `Administrator`    | Unrestricted access                                         | Break-glass accounts (MFA gated) |

### Resource Pools

Resource pools group related VMs and storage objects and are the primary unit of delegation in CHIP. Each team or project is assigned its own pool.

```bash
# List all resource pools
pvesh get /pools

# Create a new resource pool
pvesh create /pools --poolid pool-teamname --comment "Team Name workloads"

# Assign a VM to a pool
pvesh set /pools/pool-teamname --vms 205,206,207
```

<Accordion title="Granting a user access to a resource pool">
  1. In the web UI, go to **Datacenter → Permissions → Add → User Permission**.
  2. Set **Path** to `/pool/pool-teamname`.
  3. Set **User** to the target account (e.g., `jsmith@chip-ldap`).
  4. Set **Role** to `PVEVMUser` for read/console access or `PVEVMAdmin` for full control.
  5. Leave **Propagate** checked so the permission cascades to all VMs and storage within the pool.
  6. Click **Add** to save. The user will see the pool's resources on their next login.
</Accordion>

<Accordion title="Auditing current permissions">
  ```bash
  # Dump all ACL entries cluster-wide
  pvesh get /access/acl

  # Filter for entries on a specific pool
  pvesh get /access/acl | jq '.[] | select(.path | startswith("/pool/pool-teamname"))'
  ```
  Review ACLs quarterly as part of the CHIP access review cycle. Remove stale entries for departed team members promptly to comply with least-privilege requirements.
</Accordion>
